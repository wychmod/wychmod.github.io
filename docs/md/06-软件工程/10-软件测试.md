## 一、核心主题概述

软件测试是验证程序正确性、发现缺陷、评估质量并为发布决策提供依据的系统工程。本文围绕**测试基础理论**、**用例设计与执行**、**白盒与自动化进阶**、**全链路质量体系建设**、**JaCoCo 代码覆盖率**、**Chrome 插件与浏览器提效**六个维度，对归档笔记进行结构化整理。

> 💡 补充：归档中除 Markdown 笔记外，还包含可直接复用的 [`测试用例.csv`](../archive/old-testing-notes/测试用例.csv) 与 [`测试计划.doc`](../archive/old-testing-notes/测试计划.doc)，适合作为团队模板落地。

## 二、测试基础

### 2.1 软件与测试的本质

> 📚 完整资料：[测试基础/软件测试基础知识.md](../archive/old-testing-notes/测试基础/软件测试基础知识.md)

- **软件十大特性**：无形性、智能性、开发个体性、质量不可完美性、复制简单性、架构依赖性、环境依赖性、维护升级性、难以废弃性、应用广泛性。
- **软件分类**：系统软件（操作系统、数据库、语言程序、服务程序）与应用软件（浏览器、支付工具等）。
- **软件生命周期**：从计划开发到报废的全过程，按阶段拆分为可管理的任务与标准文档。
- **开发模型**：
  - 瀑布模型：线性推进，错误修正成本高；
  - 原型模型：与客户紧密迭代；
  - 螺旋模型：计划 → 风险分析 → 实施工程 → 客户评估，循环推进；
  - 敏捷模型：以人为核心、短周期迭代、增量交付。

### 2.2 测试原则与策略

- 所有测试追溯到用户需求；尽早启动测试；遵循 Pareto 法则（80% 缺陷在早中期发现，剩余 20% 中大部分由系统测试发现）；穷尽测试不可能；注意“杀虫剂怪事”，需持续更新用例；每次修复后需回归原有全部用例。
- 测试策略要素与测试过程需结合需求规格说明书检查列表落地。

### 2.3 测试用例设计方法

> 📚 完整资料：[测试基础/测试用例设计方法.md](../archive/old-testing-notes/测试基础/测试用例设计方法.md)

| 方法 | 核心思想 | 适用场景 |
|---|---|---|
| **等价类划分** | 输入域划分为有效/无效等价类，每类选代表 | 取值范围、必须成立条件、离散可选值 |
| **边界值法** | 取边界及略超边界的值 | 范围校验、越界检测 |
| **因果图法** | 描述输入条件与输出结果的逻辑关系 | 多条件组合、相互制约 |
| **判定表法** | 条件桩 + 动作桩 + 条件项 + 动作项 | 复杂条件组合决策 |
| **正交实验法** | 用正交表挑选有代表性的试验点 | 多因素多水平组合爆炸 |
| **场景法** | 基本流 + 备选流 | 用户端到端流程 |

### 2.4 执行用例与缺陷管理

> 📚 完整资料：[测试基础/执行测试用例.md](../archive/old-testing-notes/测试基础/执行测试用例.md)

**执行流程**：预置条件检查 → 执行测试步骤 → 记录实际结果 → 对比预期结果 → 标记 Pass / Fail / Blocked。

**准入标准**：
1. 开发编码结束并完成单元测试；
2. 需求功能已实现，未实现部分明确测试范围；
3. 集成测试通过，基本流程可走通；
4. 代码评审通过并符合编码规范；
5. 兼容性、安全、性能测试范围明确。

**准出标准**：用例评审通过并全部执行；覆盖率达标；缺陷录入系统；一二级错误修复率 100%，三四级 95%；遗留问题有方案；性能、兼容性、安全性达标；产出系统测试总结报告。

**缺陷报告 5C 原则**：Correct、Clear、Concise、Complete、Consistent。

> 💡 补充：缺陷标题建议采用「执行完 A 后，发生 B」或「发生 B，当 A 执行完后」的格式，避免模糊词汇并加关键字方便检索。

## 三、测试进阶

> 📚 完整资料：[测试基础/测试进阶提升.md](../archive/old-testing-notes/测试基础/测试进阶提升.md)

### 3.1 白盒测试

- **代码审查**：业务逻辑、算法效率、代码风格、编程规则、内存管理、性能优化（减少对象创建、减少循环体代码、优化异常处理、减少 I/O）。
- **逻辑覆盖**：语句覆盖 → 判定覆盖 → 条件覆盖 → 判定-条件覆盖 → 条件组合覆盖，覆盖强度逐级递增。

### 3.2 自动化测试

- 意义：缩短周期、提升效率、节省人力、增强稳定性与精确度。
- 主流工具：QTP/UFT、Selenium WebDriver、LoadRunner、Sikuli、Appium 等。
- Selenium Java 示例依赖：

```xml
<dependencies>
    <dependency>
        <groupId>org.seleniumhq.selenium</groupId>
        <artifactId>selenium-java</artifactId>
        <version>4.10.0</version>
    </dependency>
    <dependency>
        <groupId>org.testng</groupId>
        <artifactId>testng</artifactId>
        <version>7.4.0</version>
        <scope>test</scope>
    </dependency>
    <dependency>
        <groupId>io.cucumber</groupId>
        <artifactId>cucumber-java</artifactId>
        <version>7.12.0</version>
    </dependency>
    <dependency>
        <groupId>io.qameta.allure</groupId>
        <artifactId>allure-testng</artifactId>
        <version>2.22.2</version>
    </dependency>
</dependencies>
```

### 3.3 安全测试

- WASC 六大威胁分类：Authentication、Authorization、Client-Side Attacks、Command Execution、Information Disclosure、Logical Attacks。
- 可借助 APPSCAN 等安全审计工具提升效率，但需人工复核避免误报。

### 3.4 性能测试

- 目标：评估生产准备状态、判定性能基准、对比系统配置、定位瓶颈、系统调优、确定吞吐量。
- 分类：负载测试、压力测试、容量测试、配置测试、并发测试、可靠性测试、稳定性测试。
- LoadRunner 核心组件：虚拟用户脚本生成器、压力产生器、用户代理、压力调度、监视系统、结果分析工具。

### 3.5 手机 App 测试

- **Appium**：开源跨平台框架，支持 iOS/Android/FirefoxOS，Android 4.1+ 使用 UIAutomator。
- **Monkey**：Android SDK 自带工具，生成伪随机事件流，用于压力测试。

## 五、JaCoCo 与代码覆盖率

> 📚 完整资料：[jacoco/jacoco工具.md](../archive/old-testing-notes/jacoco/jacoco工具.md)

### 5.1 插桩原理

JaCoCo 通过 probe 探针注入实现覆盖率统计，支持两种模式：

- **on-the-fly 模式**：通过 `javaagent` 在类加载时动态插桩，无需提前处理，最简单常用。
- **offline 模式**：测试前预先插桩，再执行插桩后的 class/jar，适合不支持 javaagent、无法设置 JVM 参数、字节码需转换或与其他 agent 冲突的场景。

### 5.2 Maven 集成示例

```xml
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.11</version>
    <executions>
        <execution>
            <goals>
                <goal>prepare-agent</goal>
            </goals>
        </execution>
        <execution>
            <id>report</id>
            <phase>test</phase>
            <goals>
                <goal>report</goal>
            </goals>
        </execution>
    </executions>
</plugin>
```

查看报告：

```bash
mvn clean test
# 报告路径: target/site/jacoco/index.html
```

### 5.3 覆盖率指标

| 指标 | 含义 |
|---|---|
| 行覆盖 | 至少执行一次的代码行 |
| 分支覆盖 | if/else 等分支都被执行 |
| 方法覆盖 | 至少被调用一次的方法 |
| 类覆盖 | 至少被加载执行的类 |

### 5.4 增量覆盖与命令行用法

- **增量覆盖**：统计两次提交之间新增或修改代码的覆盖情况。
- 启动带 JaCoCo agent 的服务：

```bash
java -javaagent:jacocoagent.jar=includes=*,output=tcpserver,port=6300,address=localhost,append=true -jar demo-0.0.1.jar
```

- dump 生成 exec 文件：

```bash
java -jar cli-0.8.7.jar dump --address 127.0.0.1 --port 6300 --destfile jacoco-demo.exec
```

- 生成 HTML/XML 报告：

```bash
java -jar cli-0.8.7.jar report jacoco-demo.exec \
  --classfiles D:\xxx\target\classes \
  --sourcefiles D:\xxx\src\main\java \
  --html html-report --xml report.xml --encoding=utf-8
```

> 💡 补充：增量报告可结合 `cli-0.8.7-diff.jar` 与 `code-diff.jar` 生成，将 Git diff 信息以 JSON 形式传入 `--diffCode` 参数，实现仅对新变更代码的覆盖率门禁。

## 六、Chrome 插件与提效

### 6.1 测试助手 Chrome 插件设计

> 📚 完整资料：[chrome插件设计/测试助手chrome插件设计文档.md](../archive/old-testing-notes/chrome插件设计/测试助手chrome插件设计文档.md)

技术方案采用 Manifest V3 + Plasmo 框架，以 JSON 工具为核心场景，包含：

- **核心功能**：JSON 输入/输出、序列化与反序列化、实时校验与纠错、转义/反转义；
- **体验增强**：Monaco/CodeMirror 编辑器、行号/折叠/括号匹配、树形视图、暗黑模式、历史记录、一键复制；
- **高级扩展**：数据排序/过滤/模拟、Base64、时间戳转换、JSON Schema 校验、大文件分块、XSS 过滤（DOMPurify）、Service Worker 离线缓存。

Manifest V3 示例：

```json
{
  "manifest_version": 3,
  "name": "Test Helper",
  "version": "1.0",
  "permissions": ["activeTab", "storage"],
  "host_permissions": ["<all_urls>"],
  "background": {
    "service_worker": "background.js"
  },
  "content_scripts": [{
    "matches": ["<all_urls>"],
    "js": ["content.js"]
  }]
}
```

常用 Chrome API：

- `chrome.tabs` — 标签页操作
- `chrome.storage` — 本地存储
- `chrome.runtime` — 消息传递
- `chrome.contextMenus` — 右键菜单

### 6.2 网页标签管理插件

> 📚 完整资料：[chrome插件设计/网页标签管理设计文档.md](../archive/old-testing-notes/chrome插件设计/网页标签管理设计文档.md)

### 6.3 Chrome 浏览器测试提效技巧

> 📚 完整资料：[chrome浏览器使用技巧/chrome浏览器使用技巧.md](../archive/old-testing-notes/chrome浏览器使用技巧/chrome浏览器使用技巧.md)

## 七、2026 年测试工程趋势

结合归档内容与当前工程实践，软件测试正向以下方向演进：

1. **左移与右移并行**：单元测试、契约测试继续左移，生产可观测性、混沌工程、影子流量右移。
2. **AI 辅助测试生成**：基于需求文档或现有代码自动生成测试用例、填充测试数据、修复脆弱断言。
3. **精准测试与增量覆盖**：以 JaCoCo 增量覆盖、代码 diff 驱动用例推荐，减少无效回归。
4. **统一质量门禁**：将单元、接口、UI、安全、性能、覆盖率统一纳入 CI/CD Pipeline。
5. **浏览器端可编程提效**：Chrome 插件、Recorder、DevTools Protocol 成为测试工程师日常工具链。

> 💡 补充：2026 年团队应优先建立“可重复、可度量、可追踪”的测试基线，再逐步引入 AI 与精准测试，避免为追新而增加维护成本。

## 八、常见坑与补充

1. **覆盖率≠质量**：高覆盖率不能保证无缺陷，低覆盖率通常意味着风险。覆盖率应作为辅助指标而非唯一准出标准。
2. **自动化用例脆弱**：过度依赖页面 XPath、未做等待策略、未隔离测试数据，会导致用例频繁失败。
3. **忽略浏览器兼容性**：测试计划需明确目标浏览器及版本，国产浏览器（360、搜狗）与 IE 兼容仍需关注。
4. **缺陷报告不规范**：缺少复现步骤、环境信息、截图，会大幅增加开发定位成本。
5. **JaCoCo 与 JDK 版本匹配**：使用新版 JDK 时需注意 JaCoCo 版本兼容性，避免出现 `Unsupported class file major version` 错误。
