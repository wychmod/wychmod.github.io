# AI 编程三件套方法论

> **原文归档**：[archive/old-agent-notes/agent开发知识/](../archive/old-agent-notes/agent开发知识/)
> 包含：3 个文件（AI编程三件套方法论 / AI方法论 / claude.md模板）

## 一、核心主题概述

AI 时代程序员的核心竞争力，不再是手速或语法记忆，而是**判断力 + 驾驭 AI** 的能力。三件套方法论把这套能力工程化落地：

- **SDD（Spec-Driven Development）**：先把需求想清楚、写清楚，再让 AI 执行。
- **Harness Engineering**：给 AI 搭一个可靠的工作环境，让它不乱、不断、不出轨。
- **Loop Engineering**：让 AI 自主跑起来，最小化人工干预。

三者分别解决 **"想清楚""做对""AI 自动做"** 的问题。本文档同时融合了项目实战中的落地经验，包括如何用 `CLAUDE.md` 沉淀规范、如何组织模块化单体、如何处理 LLM 外部调用等。

## 二、时代背景：AI 改变了什么

### 2.1 软件工程的变化

| 工作 | 之前 | 之后 |
|---|---|---|
| 写一个 CRUD API | 2-3 小时 | 描述清楚，AI 10 分钟完成 |
| 写单元测试 | 1-2 小时 | Harness + AI，15 分钟 80% 覆盖率 |
| 从零搭框架 | 1-2 天 | Spec Kit，当天跑通 |
| 文档整理 | 半天 | AI 读代码自动生成，30 分钟 |
| Code Review | 看语法风格 | AI 做，人专注逻辑和架构 |

代码生成的边际成本趋近于零，但变化的是**执行方式**，不是**工程本质**。

### 2.2 没变的东西——这些才是判断力所在

- **系统设计**：模块划分、接口定义、数据流向，仍由人决策。
- **架构权衡**：一致性 vs 可用性、耦合 vs 扩展性，没有标准答案。
- **业务理解**：知道要解决什么问题，比知道怎么解决更难。
- **调试复杂问题**：分布式竞态、内存泄漏、边界 bug，AI 猜，人验证。
- **技术选型**：Spring Boot vs Quarkus、Kafka vs RocketMQ，懂底层才能选对。

> 💡 补充：AI 工具进化的速度，远快于这些判断能力被替代的速度。投资判断力比投资快捷键更保值。

### 2.3 三种程序员，三种结局

- **60% 原地踏步**：用了 AI 但没改变工作方式，效率提升 10-20%。
- **30% 工具依赖**：熟练用 AI 工具，能做小项目，遇到企业级复杂度就卡。
- **10% 能力跃迁**：改变思维和工作方式，把 AI 用在设计 + 执行双层，能主导复杂项目。

差异不在于用了什么工具，而在于**改没改变思维方式和工作方式**。

### 2.4 大项目用 AI 抓瞎的根因

不是 AI 不够强，是三个问题没解决：

1. **上下文断裂**：代码太多，AI 记不住全局。改了 A 模块，不知道 B 模块的依赖。
2. **需求表达失真**："做一个登录功能" 写出来不符合你的架构规范和安全策略。
3. **无法增量验证**：大项目不能一次写完。每步没有验证机制，错误累积，越到后面越难纠。

> 💡 补充：这三个问题不是"换更好的 AI"能解决的，需要方法论。

## 三、三件套方法论

### 3.1 SDD（Spec-Driven Development）

**核心**：先把要做的事写清楚，再让 AI 做。不是多一个步骤，是把"想清楚"变成 AI 能读懂的格式。

**流程**：

1. **先写 spec**：接口定义、约束、边界。
2. **Clarify**：AI 主动找模糊点，你回答，spec 更新。
3. **`/tasks → /implement`**：逐任务执行，每步可验证，错误不累积。

**与 TDD 的关系**：

| 方法 | 先写什么 | 解决什么问题 | 适用阶段 |
|---|---|---|---|
| SDD | 要做什么、为什么、边界 | AI 到底该做什么 | 0→1 |
| TDD | 什么结果才算正确 | AI 做得对不对 | 1→2/3 |

最好的流程是**用 SDD 收敛需求，用 TDD 锁住行为**。

**反例（没有 SDD）**：

```
"帮我做一个 Agent 模块"   → 写了，接口不对
"要能接 Spring 的"        → 改了，和数据库层冲突
"算了我自己改吧"         → 改完，AI 下次还是不记得
```

> 💡 补充：实战中的"三问裁剪法"可以帮助写 spec 前先做功能取舍：没有它产品还成立吗？做到什么程度够用？能不能一句话说清楚？

### 3.2 Harness Engineering

**核心**：给 AI 搭建可靠的工作环境，由四个支柱组成：

| 支柱 | 作用 | 落地形式 |
|---|---|---|
| **System Prompt** | 行为规范 | 项目级约定：编码风格、技术栈、安全规则 |
| **Tools** | 能力边界 | MCP 协议 + 自定义工具（文件、DB、API 调用） |
| **Context** | 上下文管理 | CLAUDE.md / AGENTS.md，让 AI 始终知道项目结构、依赖、进度 |
| **Subagents** | 并行分治 | 复杂任务拆子任务，多 Subagent 并行，主 Agent 协调 |

**CLAUDE.md 是 Harness 的核心落地载体**。一份完整的项目级上下文通常包含：

```markdown
## 项目概述
- 做什么、不做什么
- 团队规模、部署方式
- 技术栈

## 架构设计
- 应用架构（模块化单体 / 微服务）
- 代码组织规范
- 外部调用处理
- 部署与运维预期
- 缓存策略

## 数据库规范
- 通用字段、索引规则、分页规则
- 大表预判、pgvector 规范

## 接口规范
- RESTful 路径、统一响应、分页、错误码

## 行为指令
- 写代码时、改代码时、不确定时
```

代码组织要具体到 AI 能直接执行。例如分层规则：

- **Controller** 只做事：参数校验和调用 Service，不写业务逻辑、不做数据查询、不处理事务。
- **Service** 处理所有业务逻辑，包括事务管理、数据校验、业务规则。
- **Mapper** 只做数据库操作，不在 XML 里写复杂业务判断。
- **Entity** 和数据库表一一对应；**DTO** 控制暴露字段，不要把 Entity 直接返回给前端。

> 💡 补充：跨模块调用只能走目标模块的 `api/` 接口，禁止直接 import 其他模块的 `domain/` 或 `infra/` 类，循环依赖视为架构错误。

### 3.3 Loop Engineering

**核心**：让 AI 自主跑起来，最小化人工干预。本质是**设计并控制 ReAct 循环**的工程方法。

**ReAct 循环**：

```
Thought → Action → Observation
        ↻ 循环直到任务完成
```

**三个关键机制**：

- **Hooks**：在循环每步前后插入验证、日志、条件判断（pre / post 钩子）。
- **Plan Mode**：AI 先规划再执行，人工审批计划。
- **Headless Mode**：无人值守运行，接入 CI/CD pipeline。

**Claude Code 工具地图**：

| 工具 | 用途 |
|---|---|
| **CLAUDE.md / AGENTS.md** | 项目级 AI 行为规范，所有 session 共享 |
| **Skills / Slash Commands** | 可复用工作流，团队统一操作路径 |
| **Subagents** | 并行子任务执行，复杂项目分治 |
| **Hooks** | 事件驱动的自定义处理，pre / post 钩子 |
| **Plan Mode / Permission** | AI 先规划后执行，人工审批 |
| **MCP** | 外部工具接入：数据库、浏览器、第三方服务 |
| **Worktrees / Headless** | 多分支并行开发，无人值守模式接入 CI/CD |

> 💡 补充：这些不是可选项，是 Loop Engineering 的执行层。

## 四、2026 年实践建议

1. **把 CLAUDE.md / AGENTS.md 当作项目的第一行代码**  
   在写业务代码之前先写上下文规范。包含项目概述、架构设计、代码组织、接口规范、行为指令。规范越具体，AI 越稳定。

2. **SDD + TDD 组合使用**  
   0→1 阶段用 SDD 收敛需求；1→2/3 阶段用 TDD 锁住行为。不要只用其中一个。

3. **Harness 四支柱补齐再放量**  
   System Prompt、Tools、Context、Subagents 都到位后，再把复杂任务交给 AI。缺一个，Loop 都容易跑偏。

4. **LLM 调用按"慢且不稳定"设计**  
   独立线程池、熔断、Fallback、分层超时是标配。Fallback 配置示例：
   ```yaml
   hify.llm.fallback:
     openai: ollama
     claude: openai
     gemini: ollama
   ```

5. **数据库规范前置**  
   主键 BIGINT 自增、通用字段、逻辑删除、索引命名、游标分页、pgvector HNSW/ivfflat 索引，在建表时一次性写好，避免后期返工。

6. **用 Plan Mode 审计划，用 Headless 跑流水线**  
   危险操作、架构改动必须人工审批计划；重复性测试、文档生成、归档整理接入 CI/CD 无人值守。

7. **保持"架构师 + 工程团队"的角色感**  
   你是架构师，AI 是工程团队。产品边界、架构决策、技术取舍由你拍板；业务代码、接口、测试由 AI 生成，你验收。

8. **拿到 AI 输出后做三步检查：意图 → 质量 → 边界**  
   意图错了后面白查，顺序不能反。任何一行代码你都要能说清楚它在干什么。

## 五、常见坑与补充

1. **坑：把 AI 当搜索引擎用**  
   AI 给的信息不能直接信，要验证。尤其是技术选型、竞品分析、第三方 API 行为。

2. **坑：需求描述越短越好**  
   "做一个登录功能"会偏差。"基于 Spring Security JWT、支持 RBAC、token 过期 2h、登录失败 5 次锁定 15min" 才是 AI 能执行的 spec。

3. **坑：让 AI 一次做完大功能**  
   大项目必须分任务、分步骤、每步验证。用 `/tasks` 拆分，用 `/implement` 逐步执行。

4. **坑：不在项目里沉淀上下文**  
   每次对话都从头描述项目，上下文会断裂。用 CLAUDE.md / AGENTS.md 把约定沉淀下来。

5. **坑：忽视 LLM 的延迟和失败**  
   外部 LLM API 慢且不稳定，必须做线程隔离、熔断、重试、Fallback，不能裸调。

> 💡 补充：AI 革掉的是"写代码"，没革掉的是"想清楚要写什么"。程序员的核心竞争力 = 业务理解 × 架构判断 × 驾驭 AI 的方法论。工具会换，方法论不会。
