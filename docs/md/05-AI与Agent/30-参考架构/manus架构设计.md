# manus 架构设计

> **原文归档**：[archive/old-agent-notes/agent系统知识/manus架构设计.md](../../archive/old-agent-notes/agent系统知识/manus架构设计.md)

## 一、核心主题概述

manus 架构是一套**可后台运行、可动态配置**的 Agent 运行时设计，核心目标是把 LLM、工作流、Agent 通信协议（A2A）与工具服务（MCP）统一到一个可热更新、可异步执行的 FastAPI 服务中。

其设计要点可概括为：

- **配置驱动**：用 `app_config.yaml` / `.env` 管理 Agent、LLM、MCP、A2A 四类配置。
- **动态加载**：服务启动后，通过 `filelock + Pydantic` 校验并热加载配置与模块。
- **异步后台执行**：利用 `asyncio.create_task` 将任务放入后台，请求中断后任务仍可继续运行。
- **消息队列解耦**：输入/输出流（`input_stream` / `output_stream`）承担核心通信职责，Agent 循环读取消息并输出事件。

> 💡 补充：该架构图标题中写作“MoocManus Agent”，与当前市面上讨论的通用型 Manus 产品并不完全等同，更偏向一个**教学/工程示例性质**的 Agent 运行时参考实现。

## 三、关键组件

### 3.1 配置层

| 配置模块 | 管理内容 | 典型字段 |
|---------|---------|---------|
| `agent_config` | Agent 运行策略 | 自动执行开关、最大迭代步数、每次搜索返回条数 |
| `llm_config` | 大模型接入参数 | `base_url`、`api_key`、`model_name`、温度、`max_tokens` |
| `mcp_config` | MCP 服务开关 | MCP Server 列表、启用/禁用配置 |
| `a2a_config` | A2A 网络开关 | A2A 网络节点列表、启用/禁用配置 |

> 💡 补充：将配置拆成四块后，可以在不重启服务的情况下只更新某一类配置，例如仅替换 LLM 模型或临时关闭某个 MCP 工具。

### 3.2 服务核心层

`MoocManus` 是一个 FastAPI 服务，承担以下职责：

1. **启动时读取本地配置文件**，并用 `filelock` 防止并发写冲突。
2. **用 Pydantic 做类型校验**，确保配置字段符合预期。
3. **根据配置动态加载 LLM 工厂、AgentFlow、A2A、MCP 四个模块**。
4. **把 Agent 的输出写入 `output_stream`**，供上层或客户端消费。

### 3.3 功能模块层

| 模块 | 颜色示意 | 说明 |
|------|---------|------|
| LLM 模型工厂 | 橙色 | 统一封装 OpenAI SDK，对外提供统一生成接口 |
| AgentFlow | 黄色 | 编排多 Agent 协作流程，支持任务分解与状态流转 |
| A2A Networks | 紫色 | 负责与其他 Agent 节点通信，实现分布式 Agent 网络 |
| MCP Servers | 绿色 | 接入外部工具/资源服务，例如搜索、文件系统、数据库 |

### 3.4 任务执行层

任务执行层由 `Agent & Task` 与 `asyncio.create_task` 两部分组成：

- `Agent & Task`：描述**可后台执行**的 Agent 任务流。
- `asyncio.create_task`：真正创建协程任务；可通过 `input_stream` 写入消息，也可写入 `StopMessageEvent` 中断任务。

> 💡 补充：后台运行的设计让该架构适合长耗时任务，例如批量数据处理、长时间网页浏览或代码生成。

### 3.5 消息流层

| 队列 | 方向 | 作用 |
|------|------|------|
| `input_stream` | 输入 | 接收用户请求、外部消息或控制指令 |
| `output_stream` | 输出 | 输出 Agent 生成的消息、标题、工具结果等事件 |

事件类型如下：

| 事件 | 含义 |
|------|------|
| `MessageEvent` | 普通消息事件 |
| `TitleEvent` | 标题/会话主题事件 |
| `ToolResultEvent` | 工具调用结果事件 |
| `WaitEvent` | 等待/占位事件 |
| `StopMessageEvent` | 停止任务事件 |

## 四、工作流程

```text
 ┌─────────────┐
 │  加载配置    │
 └──────┬──────┘
        ▼
 ┌─────────────┐
 │ FastAPI 启动 │
 └──────┬──────┘
        ▼
 ┌─────────────────────┐
 │ 动态加载四大模块      │
 │ LLM / AgentFlow /   │
 │ A2A / MCP           │
 └──────┬──────────────┘
        ▼
 ┌─────────────────────┐
 │ 创建异步后台 Task    │
 └──────┬──────────────┘
        ▼
 ┌─────────────────────────────────────────────┐
 │ 循环执行：                                     │
 │   input_stream 读取 MessageEvent              │
 │        ↓                                      │
 │   Agent / AgentFlow 处理并调用工具            │
 │        ↓                                      │
 │   output_stream 写入各类事件                  │
 │        ↓                                      │
 │   直到任务完成或收到 StopMessageEvent         │
 └─────────────────────────────────────────────┘
        ▼
 ┌─────────────────────┐
 │ 客户端消费输出队列    │
 └─────────────────────┘
```

1. **配置加载**：服务启动时读取 `app_config.yaml` / `.env`，并通过 Pydantic 校验。
2. **模块初始化**：根据配置实例化 LLM、AgentFlow、A2A、MCP 模块。
3. **任务创建**：收到用户请求后，`MoocManus` 用 `asyncio.create_task` 创建后台任务。
4. **事件循环**：Agent 从 `input_stream` 读取事件，处理后再写入 `output_stream`。
5. **结果消费**：前端或其他服务订阅 `output_stream`，实时获取生成结果。
6. **优雅停止**：外部可通过向 `input_stream` 写入 `StopMessageEvent` 停止后台任务。

## 五、2026 年现状

到 2026 年，以 Manus 为代表的通用型 Agent 产品在生态层面发生了以下变化：

1. **多 Agent 协作成为标配**：主流 Agent 框架普遍支持 `规划 Agent + 执行 Agent + 校验 Agent` 的协作模式，单模型单次调用的方式已被边缘化。

2. **MCP 协议快速普及**：Model Context Protocol 成为连接 LLM 与外部工具的事实标准之一，大量 SaaS、数据库、浏览器自动化工具都提供了 MCP Server。

3. **A2A 协议受到关注**：Agent-to-Agent 协议让不同厂商/框架的 Agent 可以互相发现、通信和委托任务，跨系统 Agent 网络开始进入生产场景。

4. **后台异步执行成为刚需**：真实业务中的 Agent 任务往往耗时数分钟甚至更久，FastAPI + `asyncio` 或独立 Worker 队列（Celery / RQ / Temporal）成为常见方案。

5. **配置热更新趋于成熟**：filelock + Pydantic 仍被广泛使用，但更大规模的生产系统通常会把配置放到 etcd / Consul / 数据库中，配合事件监听实现真正的热更新。

> 💡 补充：如果要在 2026 年按此架构落地，建议将 FastAPI 服务只作为 API 网关，把真正的 Agent 执行逻辑放到独立 Worker 中，通过 Redis/RabbitMQ 做任务队列，以便水平扩展和失败重试。

## 六、常见坑与补充

1. **配置并发写冲突**
   - 坑：多个进程同时写 `app_config.yaml` 会导致配置损坏。
   - 解：使用 `filelock` 或把配置迁移到数据库/配置中心。

2. **异步任务生命周期管理**
   - 坑：HTTP 请求结束后协程可能意外取消，导致任务中断。
   - 解：用 `asyncio.create_task` 加上 `background_tasks` 或独立 Worker，确保请求返回后任务仍存活。

3. **消息队列阻塞**
   - 坑：`output_stream` 没有消费者时，生产者可能阻塞或内存持续增长。
   - 解：设置队列上限、超时丢弃策略或背压机制。

4. **工具异常未兜底**
   - 坑：MCP 工具超时或返回异常会直接中断 Agent 流程。
   - 解：在 AgentFlow 中统一包一层 `try/except` 与重试，并生成 `ToolResultEvent` 中的错误字段。

5. **LLM 参数频繁调整**
   - 坑：不同任务对温度、`max_tokens` 要求不同，写死在全局配置里不够灵活。
   - 解：在 `llm_config` 中保留多套 profile，AgentFlow 根据任务类型动态选择。

> 💡 补充：该图最右侧的 `Agent & Task` 与 `Asyncio.create_task` 是同一层级的两种表达——前者强调“任务流”，后者强调“实现机制”，阅读时不必理解为两个独立模块。
