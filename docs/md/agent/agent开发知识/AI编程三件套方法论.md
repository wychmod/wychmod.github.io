# AI 编程三件套：SDD + Harness Engineering + Loop Engineering

> 整理自《企业级 AI 编程实战营》导论讲义。AI 时代程序员的核心竞争力是 **判断力 + 驾驭 AI** 的能力，三件套方法论是这套能力的工程化落地。

---

## 一、时代背景：什么变了，什么没变

### 1.1 软件工程的变化

| 工作 | 之前 | 之后 |
|---|---|---|
| 写一个 CRUD API | 2-3 小时 | 描述清楚，AI 10 分钟完成 |
| 写单元测试 | 1-2 小时 | Harness + AI，15 分钟 80% 覆盖率 |
| 从零搭框架 | 1-2 天 | Spec Kit，当天跑通 |
| 文档整理 | 半天 | AI 读代码自动生成，30 分钟 |
| Code Review | 看语法风格 | AI 做，人专注逻辑和架构 |

代码生成的边际成本趋近于零。70% 的 GitHub 新代码由 AI 辅助生成，顶级工程师的开发速度提升 10x。

### 1.2 没变的东西——这些才是判断力所在

- **系统设计**：模块怎么划分、接口怎么定义、数据怎么流 → AI 执行，人决策
- **架构权衡**：一致性 vs 可用性、耦合 vs 扩展性 → 没有标准答案，需要判断
- **业务理解**：知道要解决什么问题，比知道怎么解决更难，也更值钱
- **调试复杂问题**：分布式竞态、内存泄漏、边界 bug → AI 猜，人验证
- **技术选型**：Spring Boot vs Quarkus、Kafka vs RocketMQ → 懂底层才能选对

> AI 工具进化的速度，远快于这些能力被替代的速度。

### 1.3 三种程序员，三种结局

- **60% 原地踏步**：用了 AI 但没改变工作方式，效率提升 10-20%
- **30% 工具依赖**：熟练用 AI 工具，能做小项目，遇到企业级复杂度就卡
- **10% 能力跃迁**：改变思维和工作方式，把 AI 用在设计 + 执行双层，能主导企业级复杂项目

差异不在于用了什么工具，在于**改没改变思维方式和工作方式**。

## 二、大项目用 AI 特别抓瞎的根因

不是 AI 不够强，是三个问题没解决：

1. **上下文断裂**：代码太多，AI 记不住全局。改了 A 模块，AI 不知道 B 模块的依赖，写出来的代码无法集成。
2. **需求表达失真**："做一个登录功能" 写出来不符合你的架构规范和安全策略。需求越模糊，偏差越大。
3. **无法增量验证**：大项目不能一次写完。每写一步怎么知道对不对？没有验证机制，错误累积，越到后面越难纠。

> 这三个问题不是"换更好的 AI"能解决的，需要方法论。

## 三、三件套方法论

### 3.1 SDD（Spec-Driven Development）—— 解决"想清楚"

**核心**：先把要做的事写清楚，再让 AI 做。**不是多一个步骤，是把"想清楚"变成 AI 能读懂的格式**。

**流程**：

1. **先写 spec**：接口定义、约束、边界
2. **Clarify**：AI 主动找模糊点，你回答，spec 更新
3. **`/tasks → /implement`**：逐任务执行，每步可验证，错误不累积

**反例（没有 SDD）**：

```
"帮我做一个 Agent 模块"   → 写了，接口不对
"要能接 Spring 的"        → 改了，和数据库层冲突
"算了我自己改吧"         → 改完，AI 下次还是不记得
```

### 3.2 Harness Engineering —— 解决"做对"

**核心**：给 AI 搭建可靠的工作环境，由四个支柱组成：

| 支柱 | 作用 | 落地形式 |
|---|---|---|
| **System Prompt** | 行为规范 | 项目级约定：编码风格、技术栈、安全规则 |
| **Tools** | 能力边界 | MCP 协议 + 自定义工具（文件、DB、API 调用） |
| **Context** | 上下文管理 | CLAUDE.md / AGENTS.md，让 AI 始终知道项目结构、依赖、进度 |
| **Subagents** | 并行分治 | 复杂任务拆子任务，多 Subagent 并行，主 Agent 协调 |

> 四支柱齐备，AI 才能可靠地在大项目里工作，不乱、不断、不出轨。

### 3.3 Loop Engineering —— 解决"AI 自动做"

**核心**：让 AI 自主跑起来，最小化人工干预。本质是**设计并控制 ReAct 循环的工程方法**。

**ReAct 循环**：

```
Thought → Action → Observation
        ↻ 循环直到任务完成
```

**三个关键机制**：

- **Hooks**：在循环每步前后插入验证、日志、条件判断（pre / post 钩子）
- **Plan Mode**：AI 先规划再执行，人工审批计划
- **Headless Mode**：无人值守运行，接入 CI/CD pipeline

**三件套分工**：

```
SDD                  → 解决"想清楚"
Harness Engineering  → 解决"做对"
Loop Engineering     → 解决"AI 自动做"
```

## 四、Claude Code 进阶工具地图

> 这些不是可选项，是 Loop Engineering 的执行层。

| 工具 | 用途 |
|---|---|
| **CLAUDE.md / AGENTS.md** | 项目级 AI 行为规范，所有 session 共享，上下文不断裂 |
| **Skills / Slash Commands** | 可复用工作流，一次定义到处用，团队统一操作路径 |
| **Subagents** | 并行子任务执行，复杂项目分治，主 Agent 协调 |
| **Hooks** | 事件驱动的自定义处理，pre / post 钩子插入验证和日志 |
| **Plan Mode / Permission** | AI 先规划后执行，人工审批，危险操作拦截 |
| **MCP** | 外部工具接入：数据库、浏览器、第三方服务 |
| **Worktrees / Headless** | 多分支并行开发，无人值守模式接入 CI/CD |

## 五、企业级 AI Agent OS 的典型工程结构

> 一个标准的企业级 AI Agent OS 通常按职责切分模块（以 OryxOS 为例）：

```
oryxos/
├── agent-api/        # Spring Boot 入口
├── agent-core/       # ReAct 引擎核心（Thought → Action → Observation）
├── agent-tools/      # 工具调用层（MCP + 自定义工具注册机制）
├── agent-memory/     # 上下文记忆（会话管理 + 向量检索 + 长对话）
├── agent-scheduler/  # 任务调度
├── agent-protocol/   # A2A 通信
├── agent-gateway/    # 对外 API 网关
├── agent-monitor/    # 可观测性
└── agent-infra/      # 基础设施层
```

工程关注点：

- **ReAct 引擎核心**：Thought → Action → Observation 循环
- **工具调用链**：MCP 协议 + 自定义工具注册机制
- **上下文记忆层**：会话管理 + 向量检索 + 长对话处理
- **A2A 通信协议**：Agent 间消息路由和协作机制
- **完整 CI/CD**：GitHub Actions 真实工程流程

## 六、程序员的路径变化

| 维度 | 之前 | 之后 |
|---|---|---|
| 执行 | 手撸代码 | 驾驭 AI |
| 工作量度量 | 代码量 | 思考质量 |
| 能力判定 | 会写 | 会驾驭 |

**程序员的目的没变**：比别人更快更好地实现功能；从业务需求出发设计系统；在复杂大项目中发挥判断力。
**到达路径变了**：手撸代码 = 执行 → 驾驭 AI = 执行。

---

## 七、一句话总结

> **AI 革掉的是"写代码"，没革掉的是"想清楚要写什么"。**
>
> 程序员的核心竞争力 = 业务理解 × 架构判断 × 驾驭 AI 的方法论。
> 工具会换，方法论不会。
