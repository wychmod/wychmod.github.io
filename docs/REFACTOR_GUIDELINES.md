# 内容重构工作规范

> **目的**：规定 `docs/md/` 合并/总结/归档过程中的工作方式、内容风格、改正范围与记录机制。
> **配套文档**：
> - [`REFACTOR_PLAN.md`](./REFACTOR_PLAN.md) — 做什么（目录、映射、执行步骤）
> - [`CORRECTIONS.md`](./CORRECTIONS.md) — 改了什么的台账

---

## 〇、核心原则（5 条）

1. **忠于原文**：保留作者原意，不替作者发言
2. **简单易懂**：用最少的字把事说清楚，不堆砌术语
3. **配套可补**：发现明显缺失可以补，但必须标注是"补充"
4. **错误必改**：错别字、错链、事实错误必须改
5. **改必有据**：每一次改动都进 `CORRECTIONS.md`，可追溯

---

## 一、合并 / 总结流程（4 步）

### Step 1：读懂
**必做**：把要合并的所有原文**完整读一遍**，形成"作者到底想讲什么"的心智模型。

- 不读不写
- 读的时候记笔记：哪些是核心、哪些是边角、哪些已经过期
- 边角料不进新文档（直接归档到 `archive/`）

### Step 2：搭骨架
**必做**：在 `md/` 下的新位置建草稿文件，先列目录大纲。

骨架要回答：
- 这篇文档的核心主题是什么？
- 读者读完应该收获什么？
- 章节顺序怎么排？（由浅入深 / 由主到次）

骨架不写正文，只写 `## 1. xxx` `## 2. xxx` 这种大纲。

### Step 3：填充
**规则**：

| 内容类型 | 处理方式 |
|---|---|
| 原文核心观点 | 保留并用自己的话重写 |
| 原文案例/代码 | 保留（必要时简化） |
| 原文表/图 | 保留并补全（如有缺失） |
| 原文边角 | 归档到 `archive/` |
| 明显缺失的"为什么" | 补充（标注"💡 补充"） |
| 明显缺失的对比表 | 补充（标注"💡 补充"） |
| 缺失的代码示例 | 补充（标注"💡 补充"） |
| 用户没学过的内容 | **不补**（不替用户加餐） |
| 编造的统计数据 | **不补**（不替用户背书） |

**风格要求**：
- 短句优先
- 一段一个观点
- 必要时加表格、列表、代码块
- 不用"首先/其次/最后"等套话
- 不写"在当今时代"等空话
- 标题用名词短语，不写"浅谈xxx"、"xxx初识"（用户原文里的标题如果是这样，可以保留作为引用）

### Step 4：自检 + 记录
**自检清单**：

- [ ] 原文核心观点都在吗？
- [ ] 死链修了吗？（用 grep 查 `.md` 链接）
- [ ] 错别字改了吗？
- [ ] 事实性错误改了吗？
- [ ] 过时信息标了"⚠️ 已过时" + 引导到替代吗？
- [ ] 改正都登记到 `CORRECTIONS.md` 了吗？
- [ ] 补充都标了"💡 补充"吗？

## 二、改正范围（什么改、什么不改）

### 2.1 必须改的

| 类别 | 例子 | 改正动作 |
|---|---|---|
| 错别字 | "登陆" → "登录" | 直接改 |
| 标点错误 | 中英文混用、全角半角 | 统一为中文标点 |
| 死链 | 链接到的文件不存在 | 改新路径或删 |
| 缺失图片 | `![](xxx.png)` 但图片已删 | 删引用或补图 |
| 明显事实错误 | "Python 是 1995 年发布"（实际 1991） | 改正 |
| 过时信息 | "现在用 Struts2" | 加"⚠️ 已过时"块 |
| 格式破损 | 代码块没闭合、表格错位 | 修 |

### 2.2 可以改但要登记

| 类别 | 例子 | 改正动作 |
|---|---|---|
| 混乱的章节顺序 | 把 1-2-3 改成逻辑更顺的 1-3-2 | 重排 |
| 重复段落 | 同一段意思出现 2 次 | 删一个 |
| 口语化严重 | "然后呢我们就…" | 改为书面语 |
| 标题不清晰 | "这个" "那部分" | 改为描述性标题 |

### 2.3 不要改的

| 类别 | 原因 |
|---|---|
| 用户的个人观点/吐槽 | 尊重原作者 |
| 风格不统一但能读懂 | 改了就成"我"的风格了 |
| 不影响理解的小毛病 | 改了没意义 |
| 用户没确认的"事实" | 你以为错其实是你不知道 |

### 2.4 拿不准怎么办

**默认不改**。在 `CORRECTIONS.md` 的"待确认"区登记，等用户拍板。

## 三、补充范围（什么能补、什么不能补）

### 3.1 可以补（必须标"💡 补充"）

- 原文有结论但缺推导 → 补 1-2 句推导
- 原文给了用法但没给"为什么" → 补"为什么"段
- 原文列了 3 个方案但没对比 → 补对比表
- 原文代码没注释 → 加少量关键注释
- 原文缺章节（如"最佳实践"、"常见坑"）→ 补
- 多个原文合一时，需要过渡段 → 写过渡

### 3.2 不能补

- 用户没学过的内容
- 编造的统计数据
- 编造的工具/库（除非是常识）
- 编造的最佳实践（你不知道是不是 best）
- 改写用户原意

### 3.3 补充的呈现方式

在文末或被补充处加一行：

```markdown
> 💡 补充：{一句话说明补了什么、为什么补}
```

或在侧边加：

```markdown
:::tip 补充
{补充内容}
:::
```

## 四、改正记录规范

### 4.1 每次改正必须登记

不管大改小改，每一处改正都进 [`CORRECTIONS.md`](./CORRECTIONS.md)。

### 4.2 登记字段

```markdown
### 2026-07-14 #N

- **文件**：`md/01-计算机基础/编程语言/Java/00-Java基础.md`
- **类型**：错别字 / 事实错误 / 死链 / 格式 / 风格
- **原文**：{原内容 1-2 句}
- **改正后**：{改后内容 1-2 句}
- **原因**：{为什么改}
- **影响范围**：{全文 / 章节 / 单段}
```

### 4.3 记录分类汇总

`CORRECTIONS.md` 顶部维护一个统计表：

| 类型 | 次数 |
|---|---|
| 错别字 | 0 |
| 事实错误 | 0 |
| 死链 | 0 |
| 格式 | 0 |
| 风格 | 0 |
| 补充 | 0 |
| **合计** | 0 |

### 4.4 重大改动需用户确认

**触发条件**（满足任一）：

- 整段重写
- 删除原文 ≥ 30%
- 改了可能影响原意的措辞
- 涉及用户观点/评价的句子
- 合并跨多个原文档（≥ 3 篇）时

**动作**：先写到 `_drafts/` 草稿，标 `<!-- DRAFT -->`，用户 review 后再覆盖。

## 五、特殊场景处理

### 5.1 过时技术（`08-过时技术/`）

按 [`REFACTOR_PLAN.md` 第 4 章](./REFACTOR_PLAN.md#四过时技术总结模板-) 的模板开头：

```
# {技术名}（⚠️ 已过时，仅作存档）

> ## ⛔ 重要提示：本技术已过时
> **最后更新于**：2026-07
> **原因**：{原因}
> ## 🔄 推荐替代技术
> {替代技术表}
> ## 📖 最新技术速览（2026 版）
> {200-500 字介绍当前主流方案}

---

# 以下为原内容存档

{原内容}
```

并把"原内容"中的过时描述加 `[过时] ` 前缀或删除（看上下文）。

### 5.2 课程笔记合并

- 16 篇奈学 ML/DL 笔记 → 不是简单拼接
- 提炼出主线（如：监督学习流程 / 神经网络基础 / CV 三大任务）
- 每篇主线文档 1500-3000 字为宜
- 原版整目录进 `archive/courses-奈学-ML-DL/`

### 5.3 跨语言笔记合并（如 Python 高级 + Python 高效）

- 先做主题分类（基础语法 / 高级特性 / 性能 / 工具）
- 主题内去重、保留最完整版本
- 主题间加过渡段

### 5.4 总结别人（书籍、博客）的笔记

- 标注来源（如"参考：《Python 源码剖析》第 5 章"）
- 不混进个人观点（个人观点另外开"我的实践"小节）
- 引用时给出来源

## 六、协作约定

### 6.1 草稿机制

新建目录 `_drafts/`（不进侧边栏），所有重大改动先在草稿写：

```
docs/
├── md/                  # 主线
├── archive/             # 归档
└── _drafts/             # 草稿（不进导航）
    ├── 2026-07-14-Java-合并稿.md
    └── ...
```

### 6.2 一次一个分类

不并发改多个分类，避免上下文混乱和链接互相干扰。

### 6.3 每个分类完成后

- 跑一遍 grep 检查死链
- 更新 `_sidebar.md` 和 `README.md`
- commit 一次
- 告诉用户验收

### 6.4 用户的反馈方式

如果你看到文档有问题，直接说"这里改一下：..."或者"这里原来写错了，是 xxx"。

## 七、附录：常见反面例子

### 7.1 不要这样合并 ❌

> "本文综合了 lb 算法小抄、左神算法、九章算法、左神 4 份资料。"

❌ 把多份原文简单拼接

### 7.2 应该这样合并 ✅

> "本文档基于 lb 算法小抄和左神算法笔记提炼，按主题分章节。原始资料见 archive。"

✅ 标明来源 + 提炼结构

### 7.3 不要这样补充 ❌

> "根据我多年的经验，这个算法的最优实现是 xxx。"

❌ 替用户发言

### 7.4 应该这样补充 ✅

> "💡 补充：原资料未提及时间复杂度，此处按 Big-O 标注。"

✅ 标明是补充、给依据

### 7.5 不要这样改错 ❌

> 原文："Spring 的 IOC 容器是用反射实现的。"
> 改成："Spring 的 IOC 容器是 Spring 框架的核心。"

❌ 改成了没用的废话

### 7.6 应该这样改错 ✅

> 原文："Spring 的 IOC 容器是用反射实现的。"
> 改成："Spring 的 IOC 容器是基于 Java 反射 + BeanDefinition 注册机制实现的。"

✅ 改正但不替换为废话
