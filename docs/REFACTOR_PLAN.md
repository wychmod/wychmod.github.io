## 一、目标与原则

### 1.1 博客定位
**混合用** — 自己复习为主，偶尔对外分享。
- 主线层要清晰可导航
- 归档层不喧宾夺主，但能随时回查
- 求职/分享场景下，主线层要"能拿得出手"

### 1.2 整理原则
1. **价值密度分层**：消化过的知识（主线） vs 课程原版（归档）
2. **按能力维度组织**：不按"我学过什么课"，按"我能做什么"组织
3. **相似内容合并**：多份笔记提炼成一份总结，旧的归档
4. **侧边栏节制**：7 个一级分类，每级 ≤ 3 层
5. **过时内容降权**：不删除，但强制标注 + 引导到替代

## 四、过时技术总结模板 ⭐ 重要

> **适用对象**：NLP、Electron、Hadoop-Spark、爬虫 等时效性强的技术。

每篇"过时技术"主线文档（`08-过时技术/*.md`）**必须**按以下模板开头：

```markdown
# {技术名}（⚠️ 已过时，仅作存档）

> ## ⛔ 重要提示：本技术已过时
>
> **最后更新于**：2026-07
> **原因**：{具体过时原因，如：被新框架替代、社区停止维护、生态萎缩等}
>
> ## 🔄 推荐替代技术
>
> | 旧技术 | 推荐替代 | 迁移要点 |
> |---|---|---|
> | {旧技术点1} | {新技术} | {一句话迁移建议} |
> | {旧技术点2} | {新技术} | {一句话迁移建议} |
>
> ## 📖 最新技术速览（2026 版）
>
> {用 200-500 字介绍当前主流方案是什么、为什么、生态如何}
>
> ---
>
> # 以下为原内容存档
>
> {原笔记内容}
```

**示例：以 `08-过时技术/00-NLP与自然语言处理.md` 为例**

```markdown
# NLP 与自然语言处理（⚠️ 已过时，仅作存档）

> ## ⛔ 重要提示：本技术已过时
>
> **最后更新于**：2026-07
> **原因**：传统 NLP 任务（分词、词性标注、NER、文本分类）已被 LLM 全面替代，
>         学习性价比低。除非做研究或特定垂直场景，否则不建议深入。
>
> ## 🔄 推荐替代技术
>
> | 旧技术 | 推荐替代 | 迁移要点 |
> |---|---|---|
> | jieba / HanLP 分词 | 直接用 LLM（GPT/Claude/Qwen） | 准确率更高，无需维护词库 |
> | 传统文本分类（TF-IDF + SVM） | Embedding + 零样本分类 | BGE/Qwen Embedding 即可 |
> | 命名实体识别（CRF） | LLM NER 抽取 | prompt 工程即可解决 80% 场景 |
> | Word2Vec / GloVe | BGE / M3E / Qwen Embedding | 上下文感知，效果更好 |
>
> ## 📖 最新技术速览（2026 版）
>
> 当前 NLP 主流是 **LLM + RAG + Agent** 路线：Embedding 模型做语义检索，
> LLM 做生成/理解，Agent 编排多步任务。传统 NLP 流水线已不是求职热点。
>
> ---
>
> # 以下为原内容存档
>
> （原奈学 NLP 笔记内容）
```

### 4.1 4 篇过时技术文档的具体替代建议

| 旧技术 | 推荐替代 | 速览要点 |
|---|---|---|
| **NLP / HanLP / jieba** | LLM + Embedding | GPT/Qwen 做理解，BGE/M3E 做检索 |
| **Electron 桌面** | Tauri / Web 化 | Tauri 体积小、性能好；多数场景 Web 化即可 |
| **Hadoop / Spark 大数据** | 云数仓 + 实时计算 | MaxCompute/Snowflake 做离线；Flink 做实时 |
| **爬虫（Scrapy 等）** | 公开 API / 数据采购 | 优先用官方 API；非爬不可时用 Playwright + 合规策略 |

| 风险 | 影响 | 回退手段 |
|---|---|---|
| 链接改错 | 部分页面打不开 | Git 回退到上一个 commit |
